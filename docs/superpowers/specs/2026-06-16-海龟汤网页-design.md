# 海龟汤网页设计文档

## 摘要
设计一个可扩展的海龟汤（ lateral thinking puzzle ）网页应用。玩家打开页面后选择一个故事，阅读“汤面”，通过对话框向 AI 提问，AI 只回答“是 / 否 / 是也不是 / 无关”。当玩家通过提问推导出所有核心线索后，页面解锁并展示“汤底”。故事数据采用 JSON 文件格式，方便后续无限添加新故事。后端使用 Python Flask 保护 Deepseek API 密钥并处理线索判定逻辑，前端使用纯 HTML/CSS/JS 实现单页应用体验。

## 关键决策
- **架构**：前端 + 轻量后端（Python Flask + 静态前端 SPA）
- **扩展性**：故事以独立 JSON 文件存放，新增故事只需复制模板并修改文字
- **线索判定**：混合模式——核心线索由故事作者预定义，Deepseek 负责判断玩家话语是否命中这些事实
- **视觉风格**：复古推理小说风，做旧纸张、深褐文字、暗红强调色
- **线索展示**：只显示已发现线索，不显示未解锁线索的内容和总数

## 实现变更

### 1. 后端（Python Flask）
暴露 3 个 JSON API：
- `GET /api/stories`：返回故事列表（含 id、title、摘要）
- `POST /api/ask`：接收 `{story_id, messages, question}`，返回 AI 的“是/否/是也不是/无关”回答
- `POST /api/check-clues`：接收 `{story_id, messages}`，返回已被玩家掌握的核心线索列表

Deepseek 调用拆分为两次：
1. 回答玩家问题（仅知道汤面，不知道汤底，避免泄露）
2. 判定已解锁线索（知道汤底和核心线索，仅用于内部判定）

### 2. 故事数据格式
```json
{
  "id": "midnight-train",
  "title": "午夜列车",
  "summary": "一名男子在午夜空无一人的列车上死去……",
  "surface": "完整汤面文字……",
  "answer": "完整汤底文字……",
  "core_clues": [
    "男子不是普通乘客",
    "车票是被他自己撕碎的",
    "死亡时间并非午夜"
  ]
}
```

### 3. 前端（HTML/CSS/JS）
- 首页：故事卡片列表
- 游戏页：
  - 汤面区：显示故事标题和汤面
  - 对话框：玩家气泡 + AI 气泡
  - 侦探笔记：只展示已解锁线索，未解锁线索不显示、不计数
  - 输入区：底部固定的提问输入框和发送按钮
  - 汤底区：全部线索解锁后显示“揭开汤底”按钮，点击展开汤底

### 4. 视觉设计
- 背景色：`#F5F0E6`（做旧纸张），加轻微噪点纹理
- 主文字色：`#3D2B1F`（深褐）
- 强调色：`#8B2635`（暗红，用于线索高亮、按钮、印章感元素）
- 字体：`Noto Serif SC`（思源宋体）作为标题和正文
- 响应式：桌面左右分栏，手机单列堆叠

### 5. 安全与错误处理
- API 密钥仅存于后端 `.env`，不暴露给前端
- 前端校验空输入和超长输入
- 后端对 Deepseek 非标准回复兜底为“无关”
- API 异常时返回友好提示：“侦探暂时无法回答，请稍后再试。”
- MVP 阶段不持久化游戏进度，刷新页面重置

## 测试计划
1. 首页能正确列出故事
2. 进入游戏后汤面正确显示
3. 提问后 AI 只返回“是 / 否 / 是也不是 / 无关”之一
4. 命中核心线索时侦探笔记新增一条；未命中时不新增
5. 所有核心线索解锁后，“揭开汤底”按钮可用并显示汤底
6. 未解锁时不泄露线索总数和内容
7. API 密钥不出现在前端源码或网络请求中
8. 手机浏览器布局正常
9. Deepseek 异常时前端显示友好错误

## 假设与默认值
- 使用 Deepseek Chat API（模型默认 `deepseek-chat`）
- 每轮调用 2 次 API（回答 + 线索判定），MVP 阶段接受此成本
- 游戏进度不保存，刷新后重新开始
- 提示系统、历史记录、用户账号等不在本次范围内，后续可扩展
- 提供 1 个内置示例故事用于验证全流程
